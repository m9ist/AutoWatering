# Задача: подобрать паттерн детекции подключённого клапана через ACS712

Дата: 2026-05-24

## Контекст

На линии питания клапанов стоит датчик тока ACS712 (пин `A2`). Уже есть базовая
функция `Pomp::checkValveConnected(id)` ([src/Pomp.h:326](../src/Pomp.h#L326)),
которая один раз включает клапан, мерит дельту тока и сравнивает с
`THRESHOLD_MA = 780`. Текущий порог не откалиброван (по комментарию рядом
дельта подключённого клапана ~131 mA, а сравнивается с 780 → всегда
DISCONNECTED).

Цель сегодня — **не править порог на глаз**, а собрать данные многократных
включений/выключений и понять, какой паттерн надёжно отличает «клапан
подключён» от «клапана нет».

## Тестовый стенд

- Slot **0** — клапан **подключён**.
- Slot **1** — клапана **нет** (висит «в воздухе»).
- Остальные слоты в этой сессии не используем.

Имея данные с этих двух соседних слотов, можно прямо сравнить распределения
и решить, по какому признаку отличать (дельта среднего, дисперсия,
N-из-M повторов и т.п.).

## Что делаем

### Идея измерения

Клапан — это соленоид. При подаче питания получаем **inrush-пик** тока в
момент открытия, дальше потребление падает. Используем сам факт пика как
критерий «клапан подключён»: на пустом слоте пика быть не должно.

Заранее не знаем ни длительности пика, ни формы кривой → **снимаем
полный профиль тока во времени** и анализируем уже на компе. На ардуине
никаких пороговых решений сегодня не принимаем.

#### Профиль одного ON/OFF цикла

- `base` — один замер `measureDcAvg(N_BASE_SAMPLES)` перед ON.
- **ON-окно**: 2 секунды с шагом 30 мс ⇒ ~67 точек.
- **TAIL-окно**: 1 секунда после OFF с шагом 30 мс ⇒ ~33 точки (смотрим
  спад; пригодится, если у пустого слота есть короткие выбросы и
  отличать придётся по форме, а не только по амплитуде).
- `GAP_BETWEEN_CYCLES_MS` — пауза перед следующим циклом.

Замеры за одну итерацию собираем в **локальные массивы int** на стеке
(~67 + 33 = ~100 элементов × 2 байта = ~200 байт). После цикла дампим
в Serial одной CSV-подобной строкой и переиспользуем буферы для
следующего цикла. На SRAM Mega2560 этого хватит без хип-фрагментации.

Грубый псевдокод одного цикла:

```
base = measureDcAvg(N_BASE_SAMPLES)
turnOnValve(id)
for k in 0..ON_SAMPLES-1:
    on[k] = ACS.mA_DC()
    wait_until(start_on + (k+1)*SAMPLE_PERIOD_MS)
    wdt_reset()
turnOffValve(id)
for k in 0..TAIL_SAMPLES-1:
    tail[k] = ACS.mA_DC()
    wait_until(start_off + (k+1)*SAMPLE_PERIOD_MS)
    wdt_reset()
dumpProbeLine(id, round, iter, base, on, tail)
delay(GAP_BETWEEN_CYCLES_MS)
```

Watchdog: текущий `WDTO_8S` ([sketch_auto_watering.ino:78](../src/sketch_auto_watering.ino#L78)).
Один цикл ≈ 2с + 1с + GAP + дамп ≈ 3.3с. `wdt_reset()` обязательно
**внутри** циклов замеров.

### Формат лога

Одна строка на цикл, чтобы потом легко распарсить:

```
VALVE_PROBE id=<id> round=<r> iter=<k> base=<mA> on=<v0>,<v1>,...,<v66> tail=<v0>,<v1>,...,<v32>
```

Сводку по слоту на ардуине считать **не нужно** — всё считаем на компе
по логу.

### Параметры (константы)

В начале метода:

- `N_BASE_SAMPLES = 10` — замеров для усреднённого baseline.
- `SAMPLE_PERIOD_MS = 30` — шаг между точками профиля.
- `ON_WINDOW_MS = 2000` ⇒ `ON_SAMPLES = ON_WINDOW_MS / SAMPLE_PERIOD_MS`.
- `TAIL_WINDOW_MS = 1000` ⇒ `TAIL_SAMPLES = TAIL_WINDOW_MS / SAMPLE_PERIOD_MS`.
- `GAP_BETWEEN_CYCLES_MS = 200` — клапан без тока закрывается быстро.
- `CYCLES_PER_PROBE = 3` — циклов ON/OFF на одну «проверку» слота.
- `ANCHOR_ROUNDS = 3` — раундов «якорь + рандом».
- `ANCHOR_SLOT = 0` — заведомо подключённый слот.

### Порядок прогона

Не просто `[0, 1]`, а с чередованием якоря и случайного слота, чтобы
исключить дрейф midpoint и сравнить подключённый клапан с
«заведомо пустым» в одних и тех же условиях:

```
повторить ANCHOR_ROUNDS раз:
    probe(ANCHOR_SLOT)            // 3 цикла ON/OFF на slot 0
    rnd = random(1, 16)           // 1..15, пустой слот на стенде
    probe(rnd)                    // 3 цикла ON/OFF
```

Итого на старте: 3 раунда × 2 слота × 3 цикла = **18 циклов** ON/OFF.
Время прогона ≈ 18 × 3.3с ≈ **60 секунд**.

### Интеграция

- **Новый публичный метод в `Pomp.h`** (рабочее имя
  `runValveProbe(AwLogging&)` — уточним при реализации).
- **Запуск** — из `loop()` в
  [sketch_auto_watering.ino:267](../src/sketch_auto_watering.ino#L267).
  Сейчас по `isCheckButtonPressed()` дёргается `sensors.tmpLoop`. Заменяем
  на вызов нового метода. `sensors.tmpLoop` пока трогать аккуратно — по
  умолчанию просто заменить вызов.
- **Помпу НЕ включаем** в этом сценарии. Только клапаны и ACS712.

## Что НЕ делаем сегодня

- Не подбираем финальный `THRESHOLD_MA` и не правим `checkValveConnected`
  по итогам одного прогона. Сначала смотрим данные, потом — отдельным
  шагом — калибруем.
- Не трогаем существующие диагностические логи в
  `startWaterPlant`/`stopWaterPlant` — они тоже про ток, но это другой
  сценарий (с помпой).
- Не выводим результаты в Telegram — пока только локальный лог через
  `AwLogging`.

## Критерий готовности

- Нажатие check-кнопки запускает прогон и в логе появляются строки с
  замерами по slot 0 и slot 1 (минимум по N=5 циклов на слот).
- По логу видно различие между подключённым (slot 0) и неподключённым
  (slot 1) клапаном — либо по средней дельте, либо по другому признаку,
  который мы сможем обсудить и взять за основу порога.
- Watchdog не срабатывает, скетч после прогона продолжает работать
  штатно.

| Слот | avg delta, mA | peak delta, mA |
|------|---------------|----------------|
| id=0 (подключён, 8 валидных итераций) | **+167 … +406** (среднее ~+300) | **+265 … +499** |
| id=5, 8, 9 (пустые, 9 итераций)        | **-205 … -108**                  | **-131 … -43** |

Запас между классами **≥ 200 mA** (худший подключённый +167 vs лучший пустой -43).

### Наблюдения

1. **Битый base в самой первой итерации:** `id=0 round=0 iter=0 base=19`. ACS не успевает прогреться после `autoMidPoint(100, 5)` в `initPomp` — первый реальный замер уже валидный (~18000–18200). Лечится «разогревом» перед началом probe.
2. **У пустых слотов on-значения часто НИЖЕ baseline на 100–200 mA.** Гипотеза: переключение бита в shift-регистре чуть «подсаживает» линию питания, через которую же течёт измеряемый ток. Контраст это не портит — наоборот увеличивает.
3. **Профиль подключённого клапана — НЕ резкий пик-и-спад.** Первая точка (t=0): +100…+200 mA. Плато t=30…300мс: +350…+500 mA. Дальше медленно спадает к +200…+300 mA и держится до 2 сек. То есть никакого короткого inrush нет — **клапан тянет постоянный ток всё время, пока открыт**. Можно мерить когда угодно в окне 30–500мс.
4. **TAIL после OFF неинформативен** — у всех слотов tail = base ± 100. Ток падает к baseline за <30мс. Окно можно выкинуть.

### Что это даёт для боевой `checkValveConnected`

- `base = measureDcAvg(10)` → `turnOnValve` → ждать ~50 мс → `active = measureDcAvg(5)` → `turnOffValve` → `delta = active - base`.
- Порог: **`delta > 150 mA`** → CONNECTED.
- Текущий `THRESHOLD_MA = 780` в [Pomp.h:327](../src/Pomp.h#L327) в 5 раз больше реального сигнала — отсюда «всегда DISCONNECTED».

Но сначала — прогон 2.

## План прогона 2

Уточнения по итогам прогона 1:

1. **Warmup ACS 2 секунды** перед первым `turnOnValve`. Лог отдельной строкой
   (`VALVE_PROBE_WARMUP samples=...`), чтобы увидеть, на какой секунде midpoint реально стабилизируется.
2. **`ON_WINDOW_MS_ANCHOR = 5000`** для anchor (slot 0) — убедиться, что потребление стабильно на дистанции 5 с.
3. **`ON_WINDOW_MS_EMPTY = 500`** для пустых слотов — экономим время, профиль и так плоский.
4. **3 разных рандомных пустых слота** между двумя anchor-проверками, выбираются без повторений из [1..15]. Цель — убедиться, что показатели одинаковые на всех слотах (нет «особенных» каналов в мультиплексере / shift-регистре).
5. `CYCLES_PER_PROBE_ANCHOR = 3`, `CYCLES_PER_PROBE_EMPTY = 1` (пустых много, повторы на каждом не нужны).
6. TAIL-окно — убрать.

Структура прогона:
```
warmup 2с
повторить ANCHOR_ROUNDS=3 раз:
    probe(ANCHOR_SLOT, ON_WINDOW_MS_ANCHOR) × CYCLES_PER_PROBE_ANCHOR
    pick 3 distinct random slots from [1..15]:
        for each: probe(slot, ON_WINDOW_MS_EMPTY) × CYCLES_PER_PROBE_EMPTY
```

Бюджет времени: warmup 2 + (3 × 5 × 3) anchor 45 + (3 × 0.5 × 3) empty 4.5 + GAP'ы ~3 + дампы ~5 ≈ **~60с**.

Размер массива samples: `max(ON_SAMPLES_ANCHOR=166, WARMUP_SAMPLES=66) = 166` int × 2 байта = 332 байта на стеке.

Прогон 1 этот эффект не показывал, потому что там autoMidPoint попал по другому фазе и фоновый ток считался baseline-ом с самого начала.

### Сводка по итерациям (без битой первой)

**Anchor (id=0), 5-секундное окно × 8 валидных итераций:**
- delta avg от base: **+130 … +200 mA**.
- delta peak: **+250 … +400 mA**.
- Профиль за 5 секунд **плоский** — никакого спада, ток стабилен от 30мс до 5000мс.
- `on[0]` (t=0) часто ниже `on[1]` (t=30) на 100–250 mA — соленоиду нужно ~30мс на выход на ток.

**Пустые слоты (3, 5, 8, 9, 10, 11), 0.5с × 9 проверок:**
- delta avg: **-205 … -73 mA**.
- Все 6 разных каналов одинаковые — **в shift-регистре/мультиплексере «особенных» слотов нет**.

### Запас разделения

Минимальная подключённая дельта avg (+130) vs максимальная пустая (-73) → **разрыв ≥ 200 mA**.
Threshold ~80–100 mA даёт надёжное разделение с большим запасом в обе стороны.

### Что это значит для боевой `checkValveConnected`

1. **Warmup ACS проблему НЕ решает** — она в линии питания, а не в датчике. Убирать warmup из боевого кода смысла нет, но и оставлять ради «разогрева ACS» тоже бессмысленно.
2. **Перед боевыми проверками нужен «фиктивный» первый ON-OFF**, чтобы поднять фоновый ток на стабильный уровень. Иначе первая боевая проверка даст delta ≈ +18000 → False Positive.
3. **Threshold = 80–100 mA** на дельту avg за окно 30–150 мс.
4. Окно ON можно сократить до **~100–150 мс** (сигнал устаканивается за 30мс, дальше плато).

### Открытый вопрос для прогона 3

Что именно триггерит «фоновые +18А» — `turnOnValve` подключённого клапана, или любой `shiftOut` / `turnOnValve` независимо от наличия клапана?

**Эксперимент:** в начале `runValveProbe` после warmup сделать **первый ON-OFF на пустом слоте** (например, фиксированный id=8), потом всё как сейчас.

- Если первая итерация пустого даст delta ≈ +18000 → эффект «прогрева» вызывается любым переключением регистра. «Фиктивный прогрев» в боевом коде можно делать на пустом слоте (или вообще на `sendNewStateValves` без открытия физического канала).
- Если первая итерация пустого даст delta ≈ 0 → эффект специфичен для реального тока через подключённый клапан. Тогда «прогрев» = реальный `checkValveConnected(<anchor>)` или подобный.

## План прогона 3

Минимальные правки в `runValveProbe`:
1. После warmup — сделать **один ON-OFF на фиксированном пустом слоте** (`PROBE_PRIMER_SLOT = 8`) с окном 0.5с. В логе помечается `round=-1`.
2. Anchor (slot 0) и пустые слоты — все по **0.5с × 1 цикл** (профиль плоский на 5с мы уже подтвердили в прогоне 2, повторно не нужно).
3. 3 раунда, в каждом: anchor + 3 разных рандомных пустых.

По данным сравним:
- profile **primer-цикла** (pure-первый ON-OFF после warmup) с profile «первой пустой проверки» из прогона 2 (там она шла после anchor) и с anchor-`iter=0` прогона 2.
- Главный вопрос: в primer (на пустом слоте) скакнёт ли фон до +18А, как в прогоне 2 скакнул на anchor.

Бюджет: warmup 2с + primer 0.5с + 3 × (anchor 0.5с + 3 пустых × 0.5с) ≈ **~10–15 секунд**.

## Итоговые выводы по экспериментам

1. **Сигнал есть и устойчиво разделяет подключённый/пустой клапан.** Между минимальной подключённой и максимальной пустой дельтой — запас ≥ 100 mA даже в худшем прогоне.
2. **Профиль подключённого клапана — плоский.** На дистанции 5 секунд никакого спада, никакого специфического inrush. Мерить можно когда угодно в окне 30–500 мс после ON.
3. **TAIL после OFF неинформативен.** Ток возвращается к baseline за <30 мс.
4. **Фоновый ток ~+18А поднимается после первого `turnOnValve` в boot-цикле и держится до ребута.** Это не ток клапана — это паразитный эффект схемы (заряд ёмкостей / гистерезис / путь через ключ). Срабатывает на ЛЮБОМ первом ON независимо от подключённости.
5. Все каналы мультиплексера/shift-регистра дают **одинаковую** «пустую» картину — особенных слотов нет.
6. Текущая [Pomp.h:326 `checkValveConnected`](../src/Pomp.h#L326) с `THRESHOLD_MA = 780` нерабочая — реальная дельта в 5+ раз меньше и порог никогда не срабатывает.

## Боевая логика (черновик)

### Параметры

- `VALVE_CHECK_THRESHOLD_MA = 50` (с запасом от минимально наблюдённой anchor-дельты +47; ниже не опускаем, чтобы не цеплять шум). Калибруем по боевым прогонам.
- `VALVE_CHECK_ON_DELAY_MS = 50` (между `turnOnValve` и измерением active).
- `VALVE_CHECK_BASE_SAMPLES = 10`, `VALVE_CHECK_ACTIVE_SAMPLES = 5`.

### Алгоритм `checkValveConnected(id)`

```
base = measureDcAvg(VALVE_CHECK_BASE_SAMPLES)        // ~100 ms
turnOnValve(id)
delay(VALVE_CHECK_ON_DELAY_MS)                       // 50 ms
active = measureDcAvg(VALVE_CHECK_ACTIVE_SAMPLES)    // ~50 ms
turnOffValve(id)
delta = active - base
return delta > VALVE_CHECK_THRESHOLD_MA
```

~250 мс на один клапан. На 16 клапанов: ~4с.

### Primer

Фон надо «прогреть» **один раз после ребута**. Кандидаты по реализации:
- **A.** При каждом запросе на проверку — игнорировать результат первого вызова `checkValveConnected`, считая его primer-ом. Минус: пользователь не получит честный отчёт по первому слоту.
- **B.** Хранить флаг `bool primed` в `Pomp`. Перед первой проверкой делать «холостой» ON-OFF на фиксированном слоте (любом — хоть `ANCHOR_SLOT = 0`, хоть условном `PRIMER_SLOT`). Минус: ~150 мс лишних на первый запрос.
- **C.** Primer прямо в `initPomp`. Плюс — primer происходит «бесшумно» во время загрузки. Минус — открываем клапан на 150 мс при boot (если он подключён и в трубе вода — будет ~пара мл протёчки).

Решение — отдельно.

### Две точки использования

**1. Ручная проверка (Telegram + кнопка).**
- Команда `ESP_COMMAND_CHECK_VALVES` уже есть ([sketch_auto_watering.ino:175](../src/sketch_auto_watering.ino#L175)) — она дёргает `pomp.checkAllActiveValves(state, logger)` и шлёт результат в Telegram.
- Кнопка `isCheckButtonPressed` тоже вызывает `checkAllActiveValves` и шлёт результат в Telegram (та же функция).
- Проверяются **только активные** клапаны (с `PLANT_IS_ON`).
- `runValveProbe` и весь диагностический обвес (циклы `for(i=0;i<20;i++)` в start/stop WaterPlant, `checkAmperage`) — удалить.

**2. Мониторинг тока во время полива.**
- НЕ проверка клапана через дельту (это разовый ручной сценарий). Это контрольный сенсор «всё ли идёт как надо», аналог датчика расхода воды.
- В `waterPlant(id, amount)` параллельно работающей помпе и открытому клапану в цикле снимаем `ACS.mA_DC()` каждые ~500 мс (`AMP_SAMPLE_EVERY_ITER = 5`).
- Каждый сырой замер — в лог отдельной строкой (`Water amp: <mA>mA`).
- В строку, возвращаемую из `waterPlant`, добавляется avg — `Amperage avg: <mA>mA`. Дальше она уходит в Telegram через `sendTelegram` в [sketch_auto_watering.ino:113](../src/sketch_auto_watering.ino#L113).
- Здесь измеряется **общий** ток в линии (фоновый +18А, плюс ток помпы, плюс ток клапана). Никаких порогов сейчас не накладываем — это сбор данных для последующей калибровки контроля.

### Primer-стратегия

- Поле `bool acsPrimed = false` в `Pomp` (сбросится только при ребуте).
- Метод `primeAcsLineIfNeeded(logger)`: при первом вызове — холостой `turnOnValve(PRIMER_SLOT) → delay 150 → turnOffValve → delay 200`, ставит флаг.
- `PRIMER_SLOT = 0` — для тестов; для боевого реально пофиг (помпа выключена, протечки практически нет).
- Вызывается в начале `checkAllActiveValves` и в начале `waterPlant` (через единый хелпер).

## Открытые вопросы (на потом, по данным)

- Нужно ли менять `SAMPLE_PERIOD_MS` (30 мс) — поймём по форме кривой
  в логе. Если пик окажется уже одного отсчёта — уменьшаем; если ничего
  интересного не происходит — увеличиваем и берём окно длиннее.
- Длина `ON_WINDOW_MS` / `TAIL_WINDOW_MS` — увидим по затуханию,
  достаточно ли.
- `GAP_BETWEEN_CYCLES_MS = 200` — если по tail видно, что ток не успел
  упасть к baseline, увеличиваем.
