# Внедрение MQTT (Dealgate) в автополив — план реализации

Дата: 2026-07-04
Продолжение исследования: [`2026-05-26_yandex_alice_integration.md`](./2026-05-26_yandex_alice_integration.md)

## Цель

Подключить ESP8266 к MQTT-брокеру **Dealgate** (`mqtt.dealgate.ru`, без TLS) и решить две задачи:

1. **Телеметрия** — публикация показаний в брокер (и в Алису):
   - температура воздуха, влажность воздуха;
   - CO₂ (датчик появится, пока готовим структуру);
   - влажность почвы по каждому растению (до 16 шт.).
2. **Команда «полей 25 мл растение №2»** — произвольный объём полива конкретного
   растения из приложения Алисы (и по возможности голосом).

## Что выяснено из доков Dealgate (исходники)

Сводка фактов с привязкой к страницам справки (скачаны 2026-07-04):

### MQTT-сервер ([help/4](https://dealgate.ru/help/4.html))
- **Host:** `mqtt.dealgate.ru` (домен, не IP). **Port:** из профиля (≈1883), без TLS.
- **Протокол:** MQTT 3.1 / 3.1.1 / 5.0 → `PubSubClient` (3.1.1) подходит.
- **Username / Password:** латиница + цифры + `_@`, задаются в профиле. Сменa пароля — сразу.
- **ClientID:** обязателен, уникален; два устройства с одинаковым ID = разрыв соединения; пустая строка запрещена; **зарезервированы** `dashboard`, `yandex`, `scenario`. Допустимые символы: латиница/цифры + `_@#$%&:-/`.
- **LastWill** и **KeepAlive** поддерживаются.
- **Топики:** UTF-8, **нельзя `$`**, не могут быть пустыми, чувствительны к регистру,
  **могут содержать пробел**, могут состоять только из `/`, без NULL.
  ⚠️ **Пространство топиков — глобальное на весь сервис** (в доке нет изоляции по юзеру).
  ⇒ обязателен уникальный префикс (например, username или случайный токен).
- **Лимиты:** 300 публикаций/мин, 300 авторизаций/мин, 6 неудачных авторизаций/мин,
  30 ошибок протокола/мин. Нарушителю режут трафик/отзывают логин. Нам с огромным запасом.

### Устройства и умения ([help/5](https://dealgate.ru/help/5.html))
- Устройство создаётся **виртуально** в кабинете, затем «привязывается» к реальному через MQTT.
- **Тип устройства** (переключатель/лампа/розетка/терморегулятор/окно,ворота,кран/шторы)
  **технически ни на что не влияет** — нужен только для голосовой семантики
  («кран открывать», «розетку включать»). Для полива берём тип **«кран»**.
- **Умения** (capabilities), по группам:
  - **Тумблер** — поля `MQTT topic` + **значение для ON** + **значение для OFF**
    (точные payload-строки задаём мы сами). ⇒ on/off команда.
  - **Регулятор** — `min`, `max`, `step`; единица зависит от типа
    (температура → °C, и т.д.). ⇒ произвольное число из диапазона.
  - **Режим** — >2 состояний, упорядоченных по номеру (порядок критичен);
    после привязки к Алисе менять нельзя.
  - **Датчик/сенсор** — только чтение; есть «Датчик MQTT» (несовместим с Алисой,
    но виден в панели Dealgate, со своим именем и единицей).
- У одного устройства может быть **несколько умений**.
- После привязки к Алисе редактирование запрещено — нужно сначала отвязать.

### JSON-формат ([help/15](https://dealgate.ru/help/15.html)) — ключевое для телеметрии
- В настройках умения можно задать **JSON key** — брокер сам извлечёт значение из
  опубликованного JSON (для датчиков) или завернёт команду.
- Поддержка массивов `[ ]` и вложенных объектов через точку: `Ds18b20.Value`, `pin[0]`,
  `[1].pressure`. ⇒ **одно сообщение-состояние может кормить сразу много умений**
  через разные JSON-ключи.

### История/графики ([help/10](https://dealgate.ru/help/10.html))
- Данные копятся в БД брокера (хранение 1 год), если топик привязан к умению.
- Графики строятся автоматически. ⇒ бесплатные графики в кабинете без своего кода.

### Сценарии ([help/12](https://dealgate.ru/help/12.html))
- Серверная визуальная автоматизация между умениями (не нужен свой код).
- Пригодится для «полей всё разом» (сценарий дёргает 16 on/off). Опционально.

## Архитектура

```
[Приложение/голос Алисы]
   │  Yandex УДЯ (навык Dealgate, без своей Cloud Function)
   ▼
[Dealgate: навык → MQTT-брокер]   mqtt.dealgate.ru:1883, без TLS
   │  MQTT 3.1.1  (PubSubClient поверх обычного WiFiClient)
   ▼
[ESP8266]  ← подписки на командные топики; публикация state в retained-топик
   │  UART JSON (существующий протокол Communication.cpp)
   ▼
[Arduino Mega 2560]  помпа, клапаны, датчики (SHT31 +未来的 CO₂ + влажность почвы)
```

Параллельные каналы остаются: **Telegram-бот** (нужен `WiFiClientSecure`) и **Gyver/OTA**.
⚠️ Telegram использует TLS — **`WiFiClientSecure` оставляем для него отдельным клиентом**,
а MQTT поднимаем на обычном `WiFiClient`. Это два разных объекта.

## Схема MQTT-топиков и пейлоадов

Префикс `aw` уникализируем юзернеймом/токеном: `<user>/aw/...`.
Ниже `BASE = "<user>/aw"`.

| Назначение | Топик | Payload | Retained | Направление |
|---|---|---|---|---|
| Состояние (всё сразу) | `BASE/state` | JSON (см. ниже) | **да** | ESP → broker |
| Online/offline | `BASE/lwt` | `ON` / `OFF` | да | LWT + on-connect |
| Полив вкл (по умолчанию) | `BASE/p/<n>/power` | `1` (ON) / `0` (OFF) | нет | broker → ESP |
| Полив N мл (объём) | `BASE/p/<n>/vol` | `<число мл>` | нет | broker → ESP |
| Проверить клапаны | `BASE/cmd/check` | `1` | нет | broker → ESP |
| Дневной полив | `BASE/cmd/daily` | `1` | нет | broker → ESP |

Payload `BASE/state` (одно сообщение кормит все датчики через JSON key):
```json
{"t":24.5,"h":48.2,"co2":612,"ram":4321,
 "p":[{"id":0,"on":1,"or":321,"m":25},{"id":2,"on":1,"or":450,"m":40}]}
```
Привязка JSON key в кабинете (примеры):
- датчик температуры → `t`; влажность воздуха → `h`; CO₂ → `co2`;
- влажность почвы растения N → `p[N].or` (или отдельный топик на растение — см. «Дизайн-решения»).

## Как решается задача 1 (телеметрия)

1. ESP получает `ARDUINO_COMMAND_STATE` от Mega по UART (уже работает,
   `processMessageArduino` в [`EspMain.cpp`](../src/EspMain.cpp)).
2. Сериализуем state в JSON (расширяем `serializeState` в [`State.h`](../src/State.h)):
   добавляем `co2`, при желании переименовать/оставить `t`,`h`.
3. Публикуем в `BASE/state` **с retained-флагом** каждый раз при свежем state,
   а также сразу после (re)connect к брокеру.
4. В кабинете Dealgate создаём **датчики** (температура/влажность/CO₂/почва N),
   каждый指向 топик `BASE/state` со своим **JSON key**.
5. Алиса при query читает последнее retained-значение → озвучивает.
6. История/графики копятся в кабинете автоматически (бонус).

⚠️ **Retained у Dealgate в доке не упомянут** — это главный риск задачи 1.
Проверяем опытно на Этапе 1. Если retained не работает — mitigation: публиковать
state по таймеру (каждые 1–5 мин) + отвечать на RPC-запрос (см. «Открытые вопросы»).

## Как решается задача 2 («полей 25 мл растение 2»)

У УДЯ `range` поддерживает instance **`volume`** ⇒ можно передать произвольное число.
Подтверждено из доки: регулятор публикует само число в свой topic.

**Рекомендуемая модель — устройство «Кран: Полив растения N» с двумя умениями:**
- **Тумблер «Питание»** → topic `BASE/p/<n>/power`, ON=`1`/OFF=`0`.
  Включение = полить на `dailyAmountMl` (стандартная норма). Голосом: «Алиса, полей цветок 2».
- **Регулятор «Объём»** (instance `volume`, 0…500 мл, step 5) → topic `BASE/p/<n>/vol`.
  Любое пришедшее число X трактуем как «полить X мл прямо сейчас» и вызываем
  `serialCommandWater(n, X)`. Приложение — точно работает; голос («установи объём… на 25»)
  проверяем опытно.

**Почему это работает технически:** на Mega `waterPlant(id, amount)` уже принимает
произвольный мл ([`sketch_auto_watering.ino:167`](../src/sketch_auto_watering.ino)),
а `serialCommandWater(id, amount)` в EspMain уже формирует нужный UART-JSON.
То есть новое — только распарсить число из MQTT и вызвать существующую функцию.

**Альтернативы (запасные):**
- Только тумблер + `mode`-пресеты «мало/средне/много» (Dealgate `mode` поддержан) —
  голос дружелюбнее, но без произвольного мл.
- Командный топик с JSON `{"plant":2,"ml":25}` — но Алиса из голоса такой payload
  не сгенерирует (только фиксированные ON/OFF, число range или выбор mode).

UX-нюанс: после полива «по объёму» регулятор в приложении останется showing значение X.
Опционально обратно публиковать туда `dailyAmountMl` как «текущее». Не блокирует MVP.

## Изменения в коде

### `src/SecretHolder.cpp`
- **Удалить:** Yandex-топики (`topic_commands/events/config/state`) и сертификаты
  `publicCert()` / `privateCert()` (TLS для IoT больше не нужен).
- **Добавить:**
  ```cpp
  const char* mqtt_host     = "mqtt.dealgate.ru";
  const uint16_t mqtt_port  = 1883;
  const char* mqtt_user     = "<из профиля>";
  const char* mqtt_pass     = "<из профиля>";
  const char* mqtt_client_id = "<уникальный, не dashboard/yandex/scenario>";
  const char* mqtt_base     = "<user>/aw";   // уникальный префикс
  ```
- **Оставить:** `ssid()`, `wifiPasswork()`, `getTelegramBotToken()`.

### `src/EspMain.cpp` (основная работа)
- **Клиенты:** переименовать существующий `WiFiClientSecure espClient` →
  `wifiClientSecure` (для Telegram). Добавить `WiFiClient wifiClient;` (для MQTT).
  `UniversalTelegramBot` пересоздать на secure-клиенте.
- **MQTT-объект:** `PubSubClient mqttClient(wifiClient);` + в `setup()`
  `mqttClient.setBufferSize(1024);` (state-JSON > дефолтных 256 байт PubSubClient!),
  `mqttClient.setServer(mqtt_host, mqtt_port);`,
  `mqttClient.setCallback(mqttCallback);`,
  `mqttClient.setKeepAlive(30);`.
- **Новые функции:**
  - `void mqttCallback(char* topic, byte* payload, unsigned int len)` — парсит topic
    (`BASE/p/<n>/power`, `BASE/p/<n>/vol`, `BASE/cmd/check`, `BASE/cmd/daily`),
    вызывает `serialCommandWater`/`serialLog(ESP_COMMAND_*)`. Парсинг `<n>` из топика.
  - `bool mqttConnect()` — `connect(clientId, user, pass, "BASE/lwt", 1, true, "OFF")`,
    после успеха подписки + публикация `BASE/lwt = ON` и текущего `BASE/state`.
  - `void mqttLoop()` — если `!mqttClient.connected()` → `mqttConnect()`; иначе
    `mqttClient.loop()`. Вызывать **первой строкой `loop()`** (нельзя допускать
    больших задержек, иначе брокер рвёт соединение).
  - `void publishState(const State&)` — собирает JSON через `serializeState`,
    `mqttClient.publish("BASE/state", json, true)`.
- **Интеграция:**
  - В `setup()` после `checkWifi()` — `setupMqtt()` (setServer/buffer/callback),
    первый `mqttConnect()`.
  - В `loop()` первой строкой — `mqttLoop();`.
  - В `processMessageArduino` по `ARDUINO_COMMAND_STATE` — `publishState(lastState)`
    (замена вызову `sendMessageToIOT`).
- **Удалить мёртвый код:** `sendMessageToIOT` (REST к IoT Core) и TLS-обвес
  `setupWifiClient` с Yandex-сертификатами. Часть `setupWifiClient`, относящуюся к
  Telegram-secure (`setInsecure`, `setBufferSizes`), перенести в отдельный
  `setupTelegramClient()`.

### `src/State.h` (и Mega-сторона)
- Добавить поле `int co2 = 0;` (ppm) в `State`; добавить в `serializeState`/`deserializeState`
  как ключ `co2`. Поднять `EEPROM_VERSION`, если CO₂ хранится в EEPROM.
- На Mega: завести чтение CO₂-датчика и заполнение `global_state.co2`
  (зависит от выбранного датчика — MH-Z19/MH-Z1311A/SCD41 по UART/SoftwareWire).

### `platformio.ini`
- `PubSubClient@^2.8` уже в `lib_deps` окружения `esp` ✓ — изменений не требуется.
- Убедиться, что окружение `esp` собирается после правок (`pio run -e esp`).

## Поэтапный план

### Этап 0. Кабинет Dealgate (без кода)
1. Профиль → выбрать порт, задать `Username`/`Password`. Записать в `SecretHolder`.
2. Придумать уникальный `mqtt_base` (юзернейм + токен), ClientID для ESP.
3. Создать виртуальные устройства:
   - **Датчики** на топик `BASE/state` с JSON key: температура `t`, влажность `h`, CO2 `co2`.
   - Несколько **кранов «Полив растения N»** (для MVP хватит 3–4 ключевых;
     все 16 раздуют каталог и могут упереться в лимит — см. «Риски»). Умения:
     тумблер `power` (ON=`1`/OFF=`0`) + регулятор `volume` `vol` (0…500, step 5).
   - Кнопки-сценарии: «Проверить клапаны» (`BASE/cmd/check`), «Дневной полив» (`BASE/cmd/daily`).
4. Привязать навык **Dealgate** в «Доме с Алисой», убедиться, что устройства видны.
5. **Опытно проверить retained**: опубликовать MQTT-клиентом (mosquitto_pub /
   MQTT Explorer) значение в `BASE/state`, закрыть клиент, запросить у Алисы —
   ответит ли? Зафиксировать результат (от него зависит стратегия публикации).
**Критерий:** в приложении Алисы переключение «крана» меняет последнее значение
в диагностике Dealgate (без железа).

### Этап 1. ESP: телеметрия (задача 1)
1. Рефакторинг клиентов: `WiFiClient` + `WiFiClientSecure` (Telegram).
2. `PubSubClient`: server/buffer/callback, `mqttConnect`, `mqttLoop` в начале `loop()`.
3. `publishState` на `ARDUINO_COMMAND_STATE` + на (re)connect.
4. LWT-топик `BASE/lwt`.
5. Удалить `sendMessageToIOT` и Yandex-TLS-обвес; `co2` пока публикуем как 0.
**Критерий:** в диагностике/панели Dealgate видны актуальные t/h/почва; в Алисе
«какая температура/влажность» озвучивает последние значения. Telegram/OTA/Gyver не сломаны.

### Этап 2. ESP: команды (задача 2)
1. `mqttCallback`: парсинг `power`/`vol`/`check`/`daily`, вызов `serialCommandWater`/`serialLog`.
2. Подписки в `mqttConnect` на `BASE/p/+/power`, `BASE/p/+/vol`, `BASE/cmd/+`.
   (PubSubClient поддерживает wildcard `+`.)
3. Проверить в приложении: тумблер → полив на `dailyAmountMl`; регулятор объёма →
   полив ровно X мл (видеть по расходу/Telegram).
4. Проверить голосом фразы для `volume` («установи объём полива … на 25 миллилитров»).
   Если не работает голосом — оставить как приложение + on/off голосом.
**Критерий:** обе исходные задачи работают: «полей цветок N» (голос) и
«полей 25 мл растение 2» (приложение, голос — по результатам проверки).

### Этап 3. CO₂ (когда появится датчик)
1. Mega: чтение датчика → `global_state.co2`.
2. `State.h`: `co2` в serialize/deserialize (если не добавили в Этапе 1), bump `EEPROM_VERSION`.
3. Dealgate: датчик CO₂ на JSON key `co2` (instance УДЯ `co2_level`, ppm).
**Критерий:** CO₂ виден в Алисе и истории.

### Этап 4. Краевые случаи и стабильность
- Reconnect при обрывах Wi-Fi/брокера (PubSubClient сам не реконнектит — `mqttLoop`).
- LWT корректно переключает статус «не в сети» в кабинете.
- Защита от частых публикаций (не чаще, чем приходит state от Mega; лимит 300/мин не бьём).
- Что если клапан не отвечает — статус в Telegram (уже есть `checkValveConnected`).
- «Полить всё» — сценарий Dealgate (опционально) или отдельная команда.
- Тест длительной работы (ночь) на утечки/reconnect.

## Риски и открытые вопросы

| Риск/вопрос | Статус | Mitigation |
|---|---|---|
| **Retained у Dealgate не подтверждён** | проверить на Этапе 0/1 | публиковать по таймеру 1–5 мин; либо RPC-топик `BASE/rq/state`, на который ESP отвечает публикацией |
| **Голосом задать `volume` (мл)** | проверить на Этапе 2 | если нет — регулятор только из приложения, голосом on/off; запасной вариант — `mode`-пресеты |
| **Лимит устройств Алисы/Dealgate** (16 кранов × 2 умения + датчики) | уточнить в `@dealgate_ru` | MVP: 3–4 ключевых растения в Алисе, все 16 — в Telegram |
| **Глобальное пространство топиков** | факт (дока) | уникальный `BASE` префикс |
| **PubSubClient max packet 256 байт** | факт | `setBufferSize(1024)` |
| **MQTT-loop голодает из-за задержек** (Telegram `getUpdates` и т.п.) | факт | `mqttLoop()` первой строкой `loop()`; longPoll у Telegram выключен |
| **CO₂-датчик ещё не выбран** | вне MVP | Этап 3, поле в `State` зарезервировать заранее |

## Чек-лист приёмки (MVP)
- [ ] ESP подключается к `mqtt.dealgate.ru`, статус online в кабинете, LWT = OFF при отключении.
- [ ] Температура/влажность воздуха актуальны в Алисе и на графике Dealgate.
- [ ] Влажность почвы (выбранные растения) видна в Алисе/кабинете.
- [ ] «Алиса, полей цветок N» → полив на `dailyAmountMl`, подтверждение в Telegram.
- [ ] Из приложения Алисы «объём = 25» для растения 2 → полив ровно 25 мл.
- [ ] «Проверить клапаны» и «дневной полив» работают из Алисы.
- [ ] Telegram-бот, OTA, Gyver не регрессировали.
- [ ] Тест ночного цикла: reconnect, нет утечек, статус online стабилен.
