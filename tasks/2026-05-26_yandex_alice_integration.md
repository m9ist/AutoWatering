# Управление автополивом через Алису

Дата: 2026-05-26

## Цель

Голосом через Алису:
- «Алиса, полей цветок 5» → запуск полива растения 5 на заранее настроенный (`Plant::dailyAmountMl`) объём.
- «Алиса, проверь клапаны» → `checkAllActiveValves`, результат озвучивается / приходит в Telegram.
- «Алиса, какая влажность у цветка 3?» → значение последнего замера `originalValue`.

## Ограничения (заданы заказчиком)

- **Нет белого IP** (NAT провайдера).
- **Нет своего сервера / VPS / Home Assistant.** Бесплатные/дешёвые cloud-сервисы допустимы.
- **TLS/шифрование НЕ требуется** — заказчик: в домашней MQTT ничего секретного не летит. Это снимает разницу между брокерами по части TLS и **упрощает ESP-код** (обычный `WiFiClient` вместо `WiFiClientSecure`/BearSSL → экономия ~16-22 КБ RAM на ESP8266 + нет зависимости TLS от точности NTP).
- Aqara Hub M2 в той же сети — для DIY-мостов через неё путей не найдено. **Не используем.**

## Выбор брокера (исследование 2026-05-26)

Оба основных кандидата — российские, MQTT 3.1.1 (совместимо с `PubSubClient`), со **встроенным навыком Алисы** (своя Cloud Function не нужна).

| Критерий | **Dealgate.ru** | WQTT.ru |
|---|---|---|
| Датчики (темп./влажн.) | ✅ тип «Датчики» в доке | ✅ подтверждено |
| range / mode (диммер/пресеты) | ✅ **оба в доке** | ⚠️ не подтверждено |
| on/off + сценарий, объём в прошивке | ✅ + Telegram-уведомления | ✅ |
| Порт / TLS | 1883 без TLS (нам ок) | TLS 8883 |
| **Цена** | ❓ нигде не опубликована | ~299₽/год (1 мес free) |
| Лимит устройств | ❓ не указан (брокер до 10k конн.) | «искусственно не ограничено» |

Отсеяны: **flespi** (геориск, оплата из РФ под вопросом), **HiveMQ/EMQX Cloud** (из РФ не оплатить). Запасной путь — **Mosquitto на VPS** (Timeweb/Selectel ~200-300₽/мес) + свой навык по шаблону [popstas/yandex-dialogs-smarthome-mqtt](https://github.com/popstas/yandex-dialogs-smarthome-mqtt), если упрёмся в каталоги обоих managed-брокеров.

**РЕШЕНИЕ: выбран Dealgate.ru.** Зарегистрировались (вход через Яндекс-логин), прайса/платёжных форм/лимитов в кабинете нет — сервис де-факто бесплатный (Telegram-канал — техподдержка, не продажи). Функционально впереди WQTT: датчики + range + mode + Telegram-уведомления задокументированы.

### Параметры подключения Dealgate (для этапа 1)

- **Host:** `mqtt.dealgate.ru`
- **Port:** из профиля (пример 1883), без TLS
- **Username / Password:** латиница + `_@`, задаются в профиле
- **ClientID:** обязателен и уникален; зарезервированы `dashboard`, `yandex`, `scenario`
- **Протокол:** MQTT 3.1, 3.1.1, 5.0 → `PubSubClient` (3.1.1) подходит
- **Топики:** не навязаны протоколом — **задаются вручную при настройке устройства в кабинете** (для тумблера: топик + payload для ON и для OFF; для датчика: топик чтения). Это гибко — формат придумываем сами.
- **Лимиты:** 300 публикаций/мин на соединение, до 10 000 соединений на порт, 6 неудачных авторизаций/мин — нам с большим запасом.
- **Retained:** в доке явно не упомянут — **проверить опытно** на этапе 1 (нужен, чтобы stateless-навык Алисы читал последнее состояние).

## Новая архитектура (вариант A — Dealgate/WQTT, без своего навыка)

```
[Алиса]
   │ voice
   ▼
[Встроенный навык брокера (Dealgate/WQTT) в «Доме с Алисой»]
   │ внутренняя кухня брокера
   ▼
[MQTT-брокер Dealgate/WQTT]   (1883 без TLS — нам ок; retained)
   │ MQTT 3.1.1
   ▼
[ESP8266]  (PubSubClient поверх обычного WiFiClient)
   │ UART JSON (уже есть)
   ▼
[Arduino Mega 2560]
   │ shift-регистры, помпа, клапаны
```

- **Команды Алисы → устройство:** Алиса дёргает «розетку/датчик» в кабинете брокера → брокер публикует в командный топик → ESP (подписан) получает, маппит на `ESP_COMMAND_*`, шлёт в Mega по UART.
- **Состояние устройства → Алиса:** ESP публикует состояние в **retained**-топики → брокер отдаёт Алисе при query.
- **Cloud Function НЕ нужна.** Маппинг устройств делается в кабинете брокера.
- Telegram-канал остаётся параллельно — не трогаем.

Резервная **архитектура B** (если каталоги обоих managed-брокеров окажутся тесными): Mosquitto на VPS + своя Cloud Function (навык УДЯ) по шаблону popstas. У VPS свой белый IP, ESP коннектится исходящим — NAT не мешает. Дороже (~2.5-3.6к₽/год) и требует писать навык.

## Схема маппинга устройства в Алису

По исследованию протокола УДЯ (типы/capabilities) и каталогов брокеров:

| Наша сущность | Представление в Алисе | Команда/запрос |
|---|---|---|
| 16 поливалок | 16 устройств тип **«Кран»** (есть в каталоге Dealgate) `on_off` «Полив цветка N» | «Алиса, полей пятый» (сценарий → ON крана 5) |
| Температура воздуха | датчик `float/temperature` | «Алиса, какая температура» |
| Влажность воздуха | датчик `float/humidity` | «Алиса, какая влажность» |
| Влажность почвы (по растению) | датчики `float/humidity`, имя «влажность почвы цветок N» | «Алиса, влажность почвы цветка 3» |
| Проверка клапанов | «розетка»/кнопка-сценарий → `ESP_COMMAND_CHECK_VALVES` | результат в Telegram |
| Дневной полив | кнопка-сценарий → `ESP_COMMAND_DAILY_TASK` | — |

**Ограничения, заложенные в схему:**
- **«Полить N мл» голосом — невозможно** (в УДЯ `range` имеет только 6 instance: `brightness/channel/humidity/open/temperature/volume`; миллилитров и кастомных единиц нет). Объём остаётся `dailyAmountMl` в прошивке, голос только триггерит полив. Произвольный объём — через Telegram-бот.
- **Отдельного instance «влажность почвы» нет** — используем `humidity`, различаем по имени устройства. Алиса семантически не отличит от влажности воздуха, но значение озвучит.
- Если 16 датчиков почвы раздувают аккаунт/лимит — вывести в Алису 2-3 ключевых, остальное смотреть в Telegram.
- Опционально: вместо чистого on/off попробовать `mode`-пресеты объёма («мало/средне/много») — **Dealgate поддерживает `mode` по доке**, WQTT — под вопросом. Это даст голосом «полей пятый по чуть-чуть». Не критично для MVP.

## План работ

### Этап 0. Выбор и регистрация на брокере (не код)

- **Написать в `@dealgate_ru`**: бесплатен ли сервис, лимиты, потолок устройств на аккаунт (нужно ~35). По ответу выбрать Dealgate (если ок) или WQTT (запасной, ~299₽/год).
- Завести аккаунт (Dealgate — вход через Яндекс-логин; WQTT — free-месяц), создать MQTT-логин (username/password).
- Узнать endpoint/порт и **схему топиков для навыка Алисы** (у каждого брокера свой формат — изучить их доку: где command-топик, где state, нужен ли retained явно).
- Создать в кабинете устройства по «Схеме маппинга» (16 розеток-поливалок + датчики + кнопки-сценарии).
- Привязать навык брокера в «Доме с Алисой», убедиться что устройства видны.

**Критерий:** в Алисе видны устройства, переключение в приложении меняет состояние в кабинете брокера (пока без железа).

### Этап 1. MQTT-клиент на ESP

- В `EspMain.cpp` поднять `PubSubClient` поверх обычного `WiFiClient` (без TLS).
- Подключиться к брокеру (endpoint/username/password из `SecretHolder`).
- Подписаться на командные топики (формат брокера).
- В callback: парсить, маппить на `ESP_COMMAND_*`, слать в Mega через существующие `serialCommandWater` / `serialLog` (как в [`procesTelegramMessage`](../src/EspMain.cpp#L286)).
- Публиковать состояние в **retained**-топики при получении `ARDUINO_COMMAND_STATE` от Mega (состояние поливалок + значения датчиков темп/влажность/почва).
- Reconnect-логика (PubSubClient сам не реконнектит).
- Удалить мёртвый `sendMessageToIOT` и `setupWifiClient` с TLS-обвесом.

**Критерий:** переключение «розетки» в приложении Алисы → ESP получает команду → Mega открывает клапан. Обратно: статус клапана и значения датчиков видны в кабинете брокера/Алисе.

### Этап 2. Маппинг команд и UX

- «Полей цветок N» = ON «розетки» N → полив на `dailyAmountMl`. (Произвольный объём через стандартную «розетку» не задать.)
- «Проверить клапаны» — отдельная «кнопка»/сценарий → `ESP_COMMAND_CHECK_VALVES`, результат в Telegram (Алиса голосом ответит лишь «выполнено»).
- «Дневной полив» — кнопка → `ESP_COMMAND_DAILY_TASK`.
- Влажность как сенсор — если каталог WQTT позволяет тип «датчик влажности», публиковать `originalValue`.

**Критерий:** все три голосовые команды из «Цели» работают.

### Этап 3. Крайние случаи

- Offline-детект: брокерный last-will/`online`-retained-топик → в кабинете WQTT статус «не в сети».
- Что если клапан не отвечает — статус в Telegram (уже есть `checkValveConnected`).
- Группа «полить всё» — средствами Алисы.

## Открытые вопросы

Закрыто исследованием:
- ✅ Типы устройств: оба брокера умеют датчики (`float`) и исполнительные устройства. Dealgate + `range`/`mode` по доке.
- ✅ Влажность почвы: через `humidity` instance, различаем по имени.
- ✅ «N мл» голосом: невозможно (ограничение УДЯ), объём в прошивке.

Закрыто на этапе 0:
- ✅ Брокер выбран — Dealgate, бесплатен де-факто.
- ✅ Параметры подключения и типы устройств известны (см. выше). Клапаны → тип «Кран».
- ✅ Топики задаются вручную в кабинете — формат придумываем сами.

Осталось:
1. **Retained у Dealgate** — проверить опытно на этапе 1 (нужен для query Алисы). Если нет — состояние придётся отдавать иначе (узнать у техподдержки).
2. **Произвольный объём** — для MVP фикс `dailyAmountMl` через on/off. Опционально позже — `mode`-пресеты «мало/средне/много» (Dealgate поддерживает `mode`).
