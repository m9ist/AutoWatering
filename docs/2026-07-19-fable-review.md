# Ревью проекта — 2026-07-19 (Claude Fable 5)

Обзор всего кода `src/` (прошивки Mega2560 и ESP8266). Косметика отфильтрована,
ниже только то, что влияет на корректность, надёжность или сопровождаемость.

---

## 1. Архитектура, модульность

### 1.1. Общая оценка

Разделение обязанностей между Mega (железо, стейт, исполнение) и ESP (внешний мир)
здравое, `State.h` как общий контракт двух прошивок — правильное решение.
`Communication` выделен в отдельный класс с очередями — тоже хорошо. Основные
претензии ниже — про хрупкость сборки и размытые границы модулей.

### 1.2. Глобальные объекты в заголовках — хрупкость single-TU

`Screen.h` определяет глобальную `lcd`, `Pomp.h` — глобальную `flowSensor` и
free-функцию `waterFlowCount()`, `State.h` — несколько non-inline функций
(`dateToString`, `serializeState`, ...). Всё работает только потому, что каждый
заголовок включается ровно в одну единицу трансляции. Любой второй include —
ошибки линковки (ODR). Ни в одном из этих файлов нет include guard
(`Pomp.h`, `Screen.h`, `Sensors.h`, `AwClock.h`, `Graph.h`). Минимум — добавить
guard'ы и пометить функции `inline`; правильнее — вынести определения в .cpp.

### 1.3. `DEBUG_LOG` определён внутри Communication.h и управляет всем проектом

`#define DEBUG_LOG` живёт в [Communication.h:9](src/Communication.h:9) и включает
отладочное логирование во всех файлах обеих прошивок. Это неочевидная связка:
человек, выключающий «лог коммуникации», выключит логи везде. Хуже того — от него
зависит функциональность, см. пункт 3.9.

### 1.4. `Pomp` — слишком много обязанностей

Класс `Pomp` управляет насосами, клапанами (сдвиговые регистры), кнопками
(мультиплексер), датчиком расхода и датчиком тока — пять независимых устройств.
Это уже мешает: замер тока и flow-сенсор пришлось «прошивать» через публичные
методы (`beginWateringAmpStats` / `sampleWateringAmpIfNeeded` / `loopFlowSensor`),
которые вызывающий код обязан дёргать в правильном порядке. Разбиение на
`Valves`, `FlowMeter`, `CurrentSensor` упростило бы и код, и покрытие
native-тестами (сейчас тестами покрыт только `Graph`).

### 1.5. Мёртвый / нереализованный код планировщика

`AwClock::runNextDayTask` и `State::nextTaskRuning` не используются нигде
(автозапуск polива по расписанию так и не реализован, readme это честно
признаёт). При этом `nextTaskRuning` сохраняется в EEPROM и содержит мусор.
Либо доделать (там же баг 3.4), либо убрать до реализации.

### 1.6. Дублирование в EspMain

- `serialCommandWater()` и `serialConfig()` — идентичные функции, отличаются
  одной константой команды.
- Парсинг `/water` и `/config` — два почти одинаковых блока с magic-смещениями
  (`substring(12, ...)` / `substring(13, ...)`), завязанными на длину слова
  команды. Один общий парсер `<cmd> plantX Yml` убрал бы оба блока и смещения.

### 1.7. Ручная правка библиотеки SdFat вместо build-флага

Обязательный шаг «поменять `SPI_DRIVER_SELECT` в исходниках SdFat» (readme)
теряется при переустановке библиотеки. `SdFatConfig.h` уважает внешнее
определение — достаточно в `platformio.ini` для mega добавить
`build_flags = -D SPI_DRIVER_SELECT=2`, и шаг исчезает.

## 2. Эффективность решений

### 2.1. Дозирование полива — open-loop, хотя есть датчик расхода

`waterPlant()` ([Pomp.h:327](src/Pomp.h:327)) отмеряет объём по времени через
константу `practicalSpeedMlInMs = 0.0077`, а показания YF-S401 только
записываются в отчёт («Real ml = ...»). Датчик уже есть и работает — логично
замкнуть контур: лить до достижения запрошенных мл по датчику, с таймаутом по
времени как аварийным пределом. Это заодно даст бесплатную диагностику
«вода не течёт» (пульсов нет при работающем насосе).

### 2.2. String-арифметика на AVR с 8 КБ RAM

Отчёты, логи и JSON собираются конкатенацией `String` — десятки реаллокаций
на heap за один полив. Постоянный мониторинг `freeRam` в коде намекает, что
с фрагментацией уже боролись. Комментарий «оборачивать в F нельзя, виснет...»
([sketch_auto_watering.ino:174](src/sketch_auto_watering.ino:174)) — почти
наверняка следствие этой же фрагментации. Для горячих мест (отчёт о поливе,
конфиг-лист) надёжнее `snprintf` в статический буфер, как уже сделано в
`logFreeRam`.

### 2.3. Копирование State по значению

`serializeState(State state)`, `deserializeState(JsonDocument doc)`,
`isDefined(Plant plant)`, `dateToString(Ds1302::DateTime now)` — всё по
значению. `State` — это ~300+ байт (16 растений), копируется на стеке при
каждой отправке стейта. На Mega со свободными ~2-4 КБ это заметно. Передавать
`const State&` / `const Plant&`.

### 2.4. Протокол UART: chunk+ack с polling-задержками

Каждые 62 байта требуют round-trip подтверждения `next`, а `readNextChunk`
опрашивает порт с шагом `delay(10)`. Отправка стейта (~5-7 чанков) — это
десятки миллисекунд задержек чистого ожидания, и всё это время `loop()`
заблокирован (см. также 3.3). Для UART между двумя чипами на одной плате
(канал практически без потерь) хватило бы newline-delimited JSON + CRC с
неблокирующим чтением; текущий handshake-протокол — самая сложная и самая
хрупкая часть системы.

### 2.5. Полная перерисовка экрана

`loopScreen` делает `fillScreen` + полный вывод при каждом обновлении стейта
(раз в минуту по таймеру датчиков) — мигание и ~десятки мс SPI-трафика.
Достаточно перерисовывать изменившиеся строки (или хотя бы не заливать фон,
пока статус не изменился).

### 2.6. Чтение мультиплексера без settle-задержки

В `loopSoilMoistureSensors` `analogRead` идёт сразу после переключения каналов
CD74HC4067 (`delay(50)` закомментирован). У ADC AVR вход высокоомный, а
ёмкостные датчики медленные — первый замер после переключения может тащить
заряд предыдущего канала. Дешёвая страховка: двойной `analogRead` (первый
выбрасываем) — 200 мкс вместо 50 мс.

## 3. Ошибки в логике, косяки

### 3.1. [Серьёзно] Нет проверки границ plantId — порча памяти

`processEspCommand` ([sketch_auto_watering.ino:161-183](src/sketch_auto_watering.ino:161)):
`plantId` из JSON используется без проверки диапазона.
`/config plant99 50ml` пройдёт валидацию ESP (`isValidInteger` проверяет только
«все символы — цифры») и на Mega выполнит
`global_state.plants[99].dailyAmountMl = amount` — запись далеко за границей
массива, порча произвольной памяти + затем `saveStateEEPROM()` зафиксирует
испорченный стейт. `/water plant99` аналогично откроет несуществующий клапан
(бит 102 в 24-битном регистре — молча уйдёт в никуда, но полив запустится с
закрытыми клапанами → давление в магистрали). Нужна проверка
`0 <= id < PLANTS_AMOUNT` на Mega (и лучше ещё на ESP).

### 3.2. [Серьёзно] Объём полива не ограничен сверху

`/water plant3 90000ml` — легальная команда: ~3 часа непрерывного полива
одного горшка. Опечатка в Telegram (лишний ноль) → затопленный подоконник.
Нужен разумный cap (например 500 мл за команду) на стороне Mega.

### 3.3. WDT 8s против блокирующего протокола связи

`readNextChunk` блокирует до 5 с (500 × 10 мс) на каждый чанк, а приём
многочанкового сообщения в `readMessageAfterCommunicationStart` крутится без
`wdt_reset` между чанками. На хорошем канале всё быстро, но при деградации
связи (шум, зависший ESP посреди передачи) два подряд таймаута по 5 с внутри
одного `communicationTick` превышают 8-секундный WDT → «загадочные» ребуты
Mega именно в моменты проблем со связью. Добавить `wdt_reset()` в цикл чтения
чанков (или уменьшить `COMMUNICATION_TIME_OUT_CTN`).

### 3.4. `normalizeDate` принимает аргумент по значению — функция-пустышка

[AwClock.h:68](src/AwClock.h:68): `void normalizeDate(Ds1302::DateTime newDate)` —
все инкременты month/day теряются при выходе. Когда `runNextDayTask` включат,
31-е число + `day++` даст day=32, дата не нормализуется, и сравнение
`isNowAfter` сломается. Должно быть `Ds1302::DateTime&`. Сейчас дремлет,
т.к. планировщик не вызывается (см. 1.5).

### 3.5. Индикатор SD на экране всегда «хорошо»: битый препроцессор

[Screen.h:75-79](src/Screen.h:75):

```cpp
#ifdef SD_TURNED_OFF
  bool screenState = true;
#elif
  bool screenState = globalState.sdInited;
#endif
```

`SD_TURNED_OFF` определён как `false` в AwLogging.h — но `#ifdef` проверяет
факт определения, а не значение, поэтому ветка `true` берётся всегда, фон
экрана не краснеет при отвале SD. Плюс `#elif` без выражения (должен быть
`#else`). В AwLogging тот же макрос используется как runtime-значение
(`if (SD_TURNED_OFF)`) — два разных способа употребления одного флага.
Учитывая, что нижний статус-бар и так рисует отдельный индикатор `SD`,
проще всего: `bool screenState = globalState.sdInited;` без препроцессора.

### 3.6. `timeOut()` сравнивает state не с той константой

[Communication.h:57](src/Communication.h:57): `if (state == COMMUNICATION_TIME_OUT)` —
это константа `10` (длительность delay), а `readNextChunk` при таймауте ставит
`state = STATE_TIMEOUT` (= 5). Условие не срабатывает никогда: все `if (timeOut(ret)) return;`
мертвы, после таймаута код идёт дальше сравнивать мусорный `ret` с токенами и
валится в `STATE_FAILED`. Итог похожий (восстановление через hellow world), но
диагностика в логе врёт, а ветка таймаута — мёртвый код. Должно быть
`state == STATE_TIMEOUT`.

### 3.7. Дырки в данных графика усредняются как 1022

`getGrapghString` ([EspMain.cpp:274](src/EspMain.cpp:274)) не передаёт в `Graph`
точки со значением `UNDEFINED_PLANT_VALUE`, но конструктор `Graph` инициализирует
весь массив именно этим значением, а `plot()` усредняет бины без проверки на
undefined. Если у растения есть пропуски в истории (включили посреди окна,
датчик дал 1023), в соответствующие бины замешивается 1022 → ложные пики на
графике и растянутая шкала Y. В `plot()` нужно пропускать undefined-значения
при усреднении (и бин без данных рисовать пробелом).

### 3.8. Медиана смещена на элемент

`PointsHoler::getMedian` ([Graph.h:242](src/Graph.h:242)):
`_graphs[start + _ids/2 + (_ids%2)]` — для 7 элементов даёт индекс 4 (пятый),
а медиана — индекс 3. Для 3 элементов — индекс 2 вместо 1. Фильтр работает,
но систематически берёт значение выше медианы. Правильно: `start + _ids/2`.

### 3.9. `/daily` и `/checkvalves` перестанут работать без DEBUG_LOG

На ESP команды пересылаются в Mega через `serialLog(ESP_COMMAND_DAILY_TASK, ...)`
([EspMain.cpp:311](src/EspMain.cpp:311)) — а всё тело `serialLog` обёрнуто в
`#ifdef DEBUG_LOG`. Убери кто-нибудь `#define DEBUG_LOG` из Communication.h —
и две боевые команды молча исчезнут. Функциональную отправку команд нужно
делать отдельной функцией, не зависящей от логирования.

### 3.10. Повторный попап с тем же текстом не показывается

`drawScreenMessage` дедуплицирует по `lastMessage`, но между двумя вызовами
экран перерисовывает `loopScreen`. Сценарий: нажали кнопку проверки клапанов,
результат «Valve check OK: plant0 OK» показался; через минуту нажали ещё раз —
результат тот же, попап не отрисуется вовсе (только пищалки/телеграм).
`lastMessage` нужно сбрасывать в `loopScreen`.

### 3.11. Смена `pompIsOn` не триггерит обновление стейта

`updatePlantsState` ([Pomp.h:185](src/Pomp.h:185)) обновляет `state.pompIsOn`,
но `wasUpdate` выставляется только по тумблерам растений. Переключение тумблера
насоса не вызовет ни отправку стейта на ESP, ни перерисовку экрана — индикатор
`pomp` на экране обновится только со следующим чужим обновлением.

### 3.12. Таймзона: RTC хранит UTC, а не местное время

`configTime("UTC+3", ...)` — во-первых, в POSIX-нотации знак инвертирован
(«UTC+3» = UTC−3 местного), для Москвы нужно `"MSK-3"`; во-вторых, это неважно,
потому что и `getTimestamp()`, и `serialTimeSynced()` используют `gmtime_r` —
в Mega и на экран уезжает чистый UTC. Часы на экране и подписи логов врут на
3 часа. Если задумано локальное время — использовать `localtime_r` и починить
TZ-строку. Заодно: в списке NTP-серверов первая строка содержит табуляцию
внутри имени хоста (`"ntp6.ntp-servers.net\t"`) — этот сервер не резолвится.

### 3.13. Висячий указатель на клиентский сертификат (дремлет)

`setupWifiClient` ([EspMain.cpp:194](src/EspMain.cpp:194)): `x509` — локальная
переменная, а `setClientRSACert` сохраняет указатель на неё; после выхода из
функции указатель висячий (privKey к тому же утекает). Пока IoT отключён
(`if (true) return;`), не стреляет, но при реанимации Яндекс-интеграции TLS
начнёт падать «загадочно». Оба объекта должны быть статическими/глобальными.

## 4. Прочее

### 4.1. Безопасность Telegram-бота: нет авторизации

`loopTelegram` принимает команды из **любого** чата и, более того, переключает
`telegramChatId` на последнего написавшего. Любой, кто найдёт бота по имени
(`@AquatoriaAutoWatering_bot` захардкожено в коде), получает управление поливом
и перехватывает отчёты. Плюс дефолтный chat id закоммичен в репозиторий.
Минимум: whitelist chat id (в `SecretHolder`), игнор всех прочих. Заодно
`espClient.setInsecure()` отключает проверку сертификата Telegram — для
домашней сети приемлемо, но стоит осознавать (токен бота ходит через TLS без
валидации сервера).

### 4.2. Лимит Telegram 4096 символов на `/graphs`

Один график — ~500 символов; при 8+ активных растениях сообщение превысит
лимит и `sendMessage` молча вернёт false. Сейчас (3-5 растений) работает.
Слать по одному графику на сообщение — заодно решится и читаемость.

### 4.3. Мёртвый код и рассинхрон мелочей

- `DATA_CHUNK_SIZE` в State.h — дубль `COMMUNICATION_DATA_CHUNK_SIZE`, не используется.
- `sendIotFrequencyInMinutes` в State — не используется (на ESP свой таймер).
- `updatedSettings` (гивер-кнопка Confirm) выставляется и никогда не читается.
- `getBearerAuthKey()` возвращает пустой `secretKey` — заглушка при выключенном IoT.
- `PIN_POMP_MAIN` инициализируется, но насос всегда `PIN_POMP_SPARE` (todo в коде;
  в readme добавлено в известные ограничения).
- `pinmap.md` ссылается на `backlog.md`, которого нет в репозитории.

### 4.4. Тесты

Покрыт только `Graph` (native env, запуск через Docker из-за MSYS2). Самые
выгодные следующие кандидаты — то, что не требует железа: парсер Telegram-команд,
`serializeState`/`deserializeState` (roundtrip), `normalizeDate` (поймал бы 3.4),
логика очередей `Communication` (через стаб Stream).

### 4.5. Обновления документации (сделано в этом ревью)

- readme: `/checkvalves` добавлен в команды и в протокол (`esp_check_valves`),
  диагностика клапанов перенесена из «в разработке» в реализованное,
  зафиксировано, что работает всегда запасной насос.

## Приоритеты

1. **3.1 + 3.2** — валидация plantId и cap объёма на Mega (порча памяти, потоп).
2. **4.1** — whitelist chat id в Telegram (чужое управление поливом).
3. **3.3** — `wdt_reset` в цикле чтения чанков (ребуты при деградации связи).
4. **3.5, 3.10, 3.11** — индикация на экране (SD, попапы, pompIsOn).
5. **2.1** — замкнуть дозирование на датчик расхода; остальное по мере.
