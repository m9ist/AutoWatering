# Ревью проекта — 2026-07-19 (glm-5.2)

Самостоятельный разбор всего `src/` (прошивки Mega2560 и ESP8266). Косметика отфильтрована.
Там, где находка совпадает с уже существующим `docs/2026-07-19-fable-review.md`, помечаю
**[подтверждено]** и при необходимости уточняю; **[новое]** — то, что в прошлом ревью не всплыло.
Пара мест проверена препроцессором/g++, чтобы не гадать.

---

## 1. Архитектура, модульность

### 1.1. Заголовки без include guard + глобальные объекты в `.h` **[подтверждено]**
`Pomp.h`, `Screen.h`, `Sensors.h`, `Graph.h`, `AwClock.h` — без `#ifndef` guard'ов.
В `State.h` guard есть, но там же лежат **non-inline** свободные функции
(`dateToString`, `serializeState`, `deserializeState`, …). Всё это собирается
только потому, что каждый `.h` включается ровно в одну единицу трансляции; любой
второй include → ODR/multiple-definition. Минимум — guard'ы + `inline`; правильно —
определения в `.cpp`. Самая «архитектурная» хрупкость проекта.

### 1.2. `DEBUG_LOG` живёт в `Communication.h` и рулит всем проектом **[подтверждено]**
`#define DEBUG_LOG` в [Communication.h:9](src/Communication.h#L9) включает логи в обеих
прошивках. Связка неочевидна (см. ещё 3.9 — там она ломает функциональность).
`DEBUG_LOG` должен быть build-flag в `platformio.ini`, а не сидеть внутри чужого хедера.

### 1.3. `Pomp` — класс-«мусоросборник» **[подтверждено]**
Один класс отвечает за насос, клапаны (74HC595), кнопки (CD74HC4067), расходомер и
ACS712 — пять независимых устройств. Симптом: вызывающий код вынужден «прошивать»
порядок вызовов через публичные `beginWateringAmpStats` / `sampleWateringAmpIfNeeded` /
`loopFlowSensor` / `beforeLoopFlowSensor`. Разбить на `Valves`, `FlowMeter`,
`CurrentSensor` — и код, и native-тесты стали бы сильно проще (сейчас покрыт только `Graph`).

### 1.4. Мёртвый планировщик `runNextDayTask` / `State::nextTaskRuning` **[подтверждено]**
Не вызывается нигде, но `nextTaskRuning` сохраняется в EEPROM (с мусором). Либо
доделать (см. баг 3.4), либо убрать до реализации.

### 1.5. Дубликаты `serialCommandWater`/`serialConfig` и парсеров `/water`/`/config` **[подтверждено]**
Две идентичные функции (отличаются константой команды) + два парсера с magic-смещениями
`substring(12,…)` / `substring(13,…)`, завязанными на длину слова. Один общий
парсер `<cmd> plantX Yml` убирает и дубль, и смещения.

### 1.6. Ручная правка `SdFat` вместо build-flag **[подтверждено]**
Шаг «поставить `SPI_DRIVER_SELECT = 2` в исходниках библиотеки» теряется при
переустановке. Достаточно `build_flags = -D SPI_DRIVER_SELECT=2` в mega-окружении.

### 1.7. Команды исполняются синхронно в `loop()` — связь умирает на время полива **[новое]**
`processEspCommand` для `esp_daily` крутит `runDailyCommand()` (полив всех активных
растений подряд), а для `esp_water` — один длинный полив. Всё это внутри обработки
одного UART-сообщения: `loop()` не возвращается в `communicationTick` минуты.
Входящий буфер UART тем временем переполняется, chunk-протокол на стороне ESP
таймаутится, входящие команды/синхронизация времени теряются. Сейчас «прокатывает»
только потому, что полив короткий и по одной команде за раз. Это и архитектурный
запах (хендлер команды = блокирующая процедура в loop), и корень нескольких багов
из блока 3.

## 2. Эффективность решений

### 2.1. Open-loop дозирование при наличии рабочего расходомера **[подтверждено]**
`waterPlant` ([Pomp.h](src/Pomp.h)) отмеряет объём через `practicalSpeedMlInMs = 0.0077`,
а показания YF-S401 идут только в отчёт. Константа захардкожена, без калибровки и
зависит от напора/трубок/конкретного клапана. Контур уже есть — лить до набора
запрошенных мл по датчику, с таймаутом по времени как аварийным пределом.
Бонусом: бесплатная диагностика «вода не течёт» (нет пульсов при работающем насосе).

### 2.2. `String`-арифметика на AVR с 8 КБ RAM **[подтверждено]**
Отчёты/JSON собираются конкатенацией `String` — десятки heap-реаллокаций за полив.
Мониторинг `freeRam` и комментарий «оборачивать в F нельзя, виснет» — симптомы той
же фрагментации. Для горячих мест надёжнее `snprintf` в статический буфер
(как уже сделано в `logFreeRam`).

### 2.3. `State`/`Plant` копируются по значению **[подтверждено]**
`serializeState(State)`, `deserializeState(...)`, `isDefined(Plant)`, `dateToString(...)`,
плюс `lastState = state` на ESP. State — ~300 байт, на стеке Mega это ощущается.
Передавать `const State&` / `const Plant&`.

### 2.4. Блокирующий chunk+ack по UART с polling-delay **[подтверждено]**
Каждые 62 байта требуют round-trip `next`, `readNextChunk` опрашивает порт с
`delay(10)` (до 5 c на чанк). Для двух чипов на одной плате (канал без потерь)
это самая сложная и хрупкая часть системы. Хватило бы newline-delimited JSON + CRC
с неблокирующим чтением.

### 2.5. Полная перерисовка экрана **[подтверждено]**
`loopScreen` делает `fillScreen` + полный вывод при каждом обновлении стейта — мигание
и SPI-трафик. Дешёвая страховка: перерисовывать только изменившиеся строки.

### 2.6. Чтение мультиплексера без settle-задержки **[подтверждено]**
В `loopSoilMoistureSensors` `analogRead` идёт сразу после переключения канала
(`delay(50)` закомментирован). Ёмкостные датчики медленные, первый замер после
переключения может тащить заряд соседнего канала. Дешёво: двойной `analogRead`
(первый выкинуть), ~200 мкс вместо 50 мс.

### 2.7. SD-лог открывается/закрывается на каждую строку **[новое]**
`AwLogging::writeln` и `logFreeRam` делают `sd.open("LOG.txt", FILE_WRITE)` → запись →
`file.close()` на **каждый** вызов. При включённом `DEBUG_LOG` это десятки SD-write'ов
за один тик `loop()` — dominated by I/O, плюс усиленный износ карты (FAT-таблица
обновляется постоянно). Нужно держать файл открытым и flush'ить по таймеру/объёму.

### 2.8. `freeRam()` вызывается слишком часто и пишет на SD **[новое]**
`logFreeRam` дёргается до/после каждой операции и через (2.7) каждый раз уходит на SD.
На AVR ещё и сам замер heap не бесплатен. Логировать RAM разумнее периодически,
а не вокруг каждого действия.

## 3. Ошибки в логике, косяки

### 3.1. [Критично] Нет проверки границ `plantId` — порча памяти **[подтверждено, уточнено]**
`processEspCommand` ([sketch_auto_watering.ino:161](src/sketch_auto_watering.ino#L161))
берёт `id` из JSON без проверки. На ESP `isValidInteger` смотрит только «все символы —
цифры», поэтому `/config plant99 50ml` проходит валидацию, а на Mega выполняется
`global_state.plants[99].dailyAmountMl = amount` — запись ~1500 байт за пределы
массива, порча произвольной памяти, и тут же `saveStateEEPROM()` фиксирует мусор.
Для `/water plant99` дополнительно: `patchValveId(99)=102`, `bitWrite(currentState, 102, HIGH)`
= `1UL << 102` — сдвиг на ≥ ширины типа, UB; на практике на AVR может установить
произвольный бит, т.е. открыть **случайный** клапан при работающем насосе. Нужна
проверка `0 <= id < PLANTS_AMOUNT` на Mega (и желательно на ESP).

### 3.2. [Серьёзно] Объём полива не ограничен сверху **[подтверждено]**
`/water plant3 90000ml` — легальная команда, ~3 часа полива одного горшка. Опечатка
в Telegram (лишний ноль) → затопленный подоконник. Разумный cap (например 500 мл)
на стороне Mega.

### 3.3. [Серьёзно] WDT 8 c против блокирующего протокола **[подтверждено, дополнено]**
`readNextChunk` блокирует до 5 с (500 × `delay(10)`), а `readMessageAfterCommunicationStart`
крутит приём чанков **без `wdt_reset`** между ними. Два таймаута подряд внутри одного
`communicationTick` → загадочный ребут Mega именно при деградации связи. Та же
картина в отправке стейта в `communicationTick`: серия `readNextChunk` подряд без
сброса WDT. Лечится `wdt_reset()` в циклах чтения (или уменьшить `COMMUNICATION_TIME_OUT_CTN`).

### 3.4. `normalizeDate` принимает аргумент по значению — функция-пустышка **[подтверждено]**
[AwClock.h:68](src/AwClock.h#L68): все инкременты теряются на выходе. Дремлет, пока
не включен планировщик (1.4). Должно быть `Ds1302::DateTime&`.

### 3.5. SD-индикатор на экране всегда «хорошо» через `#ifdef` **[подтверждено, проверено]**
[Screen.h:75](src/Screen.h#L75):

```cpp
#ifdef SD_TURNED_OFF
  bool screenState = true;
#elif
  bool screenState = globalState.sdInited;
#endif
```

`SD_TURNED_OFF` определён как `false`, но `#ifdef` проверяет факт определения, а не
значение → ветка `true` берётся всегда. Проверил препроцессором g++: `#elif` без
выражения компилируется молча, но ветка мертва. Фон экрана не краснеет при отвале SD
(нижний статус-бар при этом отдельно рисует `SD` корректно — это и спасает). Достаточно
`bool screenState = globalState.sdInited;` без препроцессора.

### 3.6. [Серьёзно] `timeOut()` сравнивает не с тем — мёртвая ветка таймаута **[подтверждено, проверено]**
[Communication.h:57](src/Communication.h#L57): `if (state == COMMUNICATION_TIME_OUT)`,
где `COMMUNICATION_TIME_OUT = 10` (длительность delay). А `readNextChunk` при таймауте
ставит `state = STATE_TIMEOUT` (= 5). Проверил: 5 ≠ 10 → `timeOut()` **всегда false**,
все `if (timeOut(ret)) return;` — мёртвый код. После таймаута исполнение идёт дальше
парсить мусорный `ret` и падает в `STATE_FAILED`. Итог похожий (восстановление через
hellow-world), но: в логе врёт диагностика («unexpected message» вместо «timeout»), и
чистый путь восстановления по таймауту не работает вообще. Должно быть
`state == STATE_TIMEOUT`.

### 3.7. Дырки в графике усредняются как `1022` **[подтверждено]**
`getGrapghString` не передаёт в `Graph` точки со значением `UNDEFINED_PLANT_VALUE`, но
конструктор инициализирует весь массив именно этим значением, а `plot()` усредняет бины
без проверки. Пропуски в истории → ложные пики и растянутая шкала Y. В `plot()` надо
пропускать undefined при усреднении (бин без данных рисовать пробелом).

### 3.8. Медиана смещена на элемент **[подтверждено]**
`getMedian`: `_graphs[start + _ids/2 + (_ids%2)]`. Для 7 элементов даёт индекс 4
вместо 3, для 3 — индекс 2 вместо 1. Фильтр работает, но систематически берёт значение
выше медианы. Правильно: `start + _ids/2`.

### 3.9. `/daily` и `/checkvalves` перестают работать без `DEBUG_LOG` **[подтверждено, проверено]**
На ESP команды пересылаются в Mega через `serialLog(ESP_COMMAND_DAILY_TASK, …)` и
`serialLog(ESP_COMMAND_CHECK_VALVES, …)` ([EspMain.cpp:311,316](src/EspMain.cpp#L311)),
а всё тело `serialLog` обёрнуто в `#ifdef DEBUG_LOG`. Убери кто-нибудь `#define DEBUG_LOG`
— и две **боевые** команды молча исчезнут. Функциональную отправку команд надо вынести
в отдельную функцию, не зависящую от логирования.

### 3.10. Повторный попап с тем же текстом не показывается **[подтверждено]**
`drawScreenMessage` дедуплицирует по `lastMessage`, но `loopScreen` этот флаг не
сбрасывает. Два нажатия «проверка клапанов» с одинаковым результатом подряд → второй
попап не отрисуется. Сбрасывать `lastMessage` в `loopScreen`.

### 3.11. Смена `pompIsOn` не триггерит обновление стейта **[подтверждено, дополнено]**
`updatePlantsState` меняет `state.pompIsOn`, но `wasUpdate` выставляется только по
тумблерам растений. Переключение тумблера насоса не вызовет `stateUpdated()`,
`saveStateEEPROM()` и `loopScreen()` — индикатор `pomp` на экране обновится лишь при
следующем чужом обновлении.

### 3.12. RTC хранит/отдаёт UTC, а не местное время **[подтверждено, проверено]**
`configTime("UTC+3", …)` (знак в POSIX инвертирован, для Москвы надо `MSK-3`), и при
этом `getTimestamp()` + `serialTimeSynced()` используют `gmtime_r` → в Mega и в
Telegram уходит чистый UTC. Экран и подписи отчётов врут на 3 часа. Дополнительно:
в строке NTP-сервера литеральная табуляция внутри имени хоста
(`"ntp6.ntp-servers.net\t"`) — этот сервер не резолвится. Лечится `MSK-3` + `localtime_r`.

### 3.13. Висячий указатель на клиентский сертификат (дремлет) **[подтверждено]**
[EspMain.cpp:194](src/EspMain.cpp#L194): `BearSSL::X509List x509(publicCert().c_str())`
— локальная переменная, `setClientRSACert(&x509, privKey)` хранит указатель; после
выхода — висячий, `privKey` (`new`) утёк. Пока IoT отключён (`if (true) return;`),
не стреляет. Оба объекта должны быть статическими/глобальными.

### 3.14. Тот же класс бага, что 3.1, на стороне ESP — нет проверки `id` **[новое]**
`deserializeState` ([State.h](src/State.h)) берёт `id` из JSON и пишет `out.plants[id]`
без проверки границ. Сейчас Arduino = доверенный отправитель, так что дремлет. Но это
та же дыра: при любом искажении протокола/фаззинге ESP падает или портит стек. Стоит
прикрыть симметрично с 3.1.

### 3.15. Ручной полив по кнопке без таймаута **[новое]**
В `loop()` цикл `while (pomp.isWaterNowButtonPressed(i))` крутится до отпускания кнопки.
Залипшая кнопка/пин → бесконечный полив (WDT при этом честно ресетится, так что ребута
не будет и никто не узнает). Авто-полив ограничен числом итераций, ручной — нет. Нужен
cap по времени как аварийный предел.

### 3.16. Входящая очередь сообщений всего из 2 элементов с тихой потерей **[новое]**
`COMMUNICATION_IN_MESSAGES_LENGTH = 2` ([Communication.h:21](src/Communication.h#L21)),
при переполнении молча перезатирается «первый». Любая короткая пачка входящих (пользователь
прожал две команды подряд + шедулер/ESP что-то прислал) → потеря команды без следа в логах.
Исходящая очередь (`OUT = 10`) чуть комфортнее, но там та же semantics тихой потери.
Минимум — увеличить `IN` и/или сигнализировать о дропах в лог.

## 4. Прочее

### 4.1. Безопасность Telegram-бота **[подтверждено, уточнено]**
`loopTelegram` принимает команды из **любого** чата и переключает `telegramChatId` на
последнего написавшего. Любой, кто найдёт бота по имени, получает управление поливом и
перехват отчётов. Уточнение по секретам:
- `telegramChatId = "-5065686553"` захардкожен в **трекаемом** `EspMain.cpp`;
- имя бота `@AquatoriaAutoWatering_bot` тоже в коде;
- `SecretHolder.cpp` (токен, wifi-пароль, сертификаты) **правильно** в `.gitignore` и
  в репозиторий не попал — это ОК. Но дефолтный chatId всё равно живёт в исходнике.
- `espClient.setInsecure()` отключает проверку сертификата Telegram — токен бота ходит
  через TLS без валидации сервера.

Минимум: whitelist chat id в `SecretHolder`, игнор всех прочих; не переназначать
`telegramChatId` автоматически.

### 4.2. Лимит Telegram 4096 на `/graphs` **[подтверждено]**
Один график ~500–600 символов; при 8+ активных растениях сообщение превысит лимит,
`sendMessage` молча вернёт false. Слать по графику на сообщение.

### 4.3. Мёртвый код и рассинхрон мелочей **[подтверждено]**
`DATA_CHUNK_SIZE` (дубль), `sendIotFrequencyInMinutes`, `updatedSettings` (выставляется,
не читается), `getBearerAuthKey` (возвращает пустой `secretKey`), `PIN_POMP_MAIN`
(всегда spare). `pinmap.md` ссылается на отсутствующий `backlog.md`.

### 4.4. Тесты **[подтверждено]**
Покрыт только `Graph` (native env, запуск через Docker из-за MSYS2). Самые выгодные
следующие кандидаты без железа: парсер Telegram-команд, `serializeState`/`deserializeState`
(roundtrip поймал бы 3.14), `normalizeDate` (поймал бы 3.4), логика очередей
`Communication` через стаб `Stream` (поймала бы 3.6 и 3.16).

### 4.5. VLA в `Graph::plot` **[новое]**
`int finalGraph[graphLength]` и `char out[graphHeight][graphLength]` — variable-length
arrays, расширение GCC, не стандарт C++. Сейчас размеры ограничены (≤40 и =10), но при
изменении констант легко поймать stack overflow без предупреждения (особенно на ESP).
Лучше `std::array`/статический буфер с `assert`-ом на размер, либо аллокация с проверкой.

### 4.6. Дисциплина `EEPROM_VERSION` **[новое]**
Коммент в `State.h`: «Изменил, обнови EEPROM_VERSION» (сейчас 5). В `State` есть
закомментированные поля/TODO. Забыл bumped → `loadStateEEPROM` грузит мусор, т.к.
`version == EEPROM_VERSION`, аlayout уже не совпал. Стоит либо статик-assert на
`sizeof(State)`, либо хранить размер рядом с version и проверять оба.

## Приоритеты

1. **3.1 + 3.2** — валидация `plantId` и cap объёма на Mega (удалённо триггеряемая порча
   памяти + потоп). Самое острое.
2. **3.6** — починить `timeOut()` (`STATE_TIMEOUT` вместо `COMMUNICATION_TIME_OUT`):
   мёртвая ветка восстановления по таймауту + врущая диагностика.
3. **3.3** — `wdt_reset` в циклах чтения чанков (загадочные ребуты при деградации связи).
4. **4.1** — whitelist chat id в Telegram (чужое управление поливом).
5. **3.9** — вынести пересылку команд из-под `#ifdef DEBUG_LOG` (боевые команды
   исчезают от одного `#define`).
6. **1.1 + 1.2** — include guard'ы + убрать `#define DEBUG_LOG` из `Communication.h`
   (фундамент для всего остального рефакторинга и тестов).
